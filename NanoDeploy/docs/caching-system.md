# 缓存系统设计文档 (Cache System V1)

## 1. 背景与动机

在当前 NanoDeploy 全面拥抱的 **PD 分离（Prefill-Decode Disaggregation）** 架构中，系统经常面临超长文本和多轮对话场景。尤其在 **MoE (专家并行)** 负载下，利用 Chunked Prefill 打散长文本是平滑显存峰值的关键。

而在这一过程中，**复用历史 KV Cache** 能够带来两大核心收益：

1. **节省物理显存**：避免相同的前缀（如 System Prompt、公共知识库）在显存中存下多份冗余。
2. **极速降低 TTFT（首字延迟）**：将计算密集的 Attention 过程转化为 O(1) 的内存指针复用。

**V1 版本设计哲学：追求“极简、稳健、可维护”。**
考虑到大模型底层调度极易在并发状态机中滋生 Bug，V1 版本的缓存系统暂不引入复杂的增量树状结构（如 Radix Tree），而是采用最保守、边界最清晰的\*\*“严格块级哈希匹配（Strict Block-Level Hash Matching）”\*\*策略，以确保系统的绝对稳定性和极低的后期维护成本。

## 2. 基本数据结构

### 2.1 块哈希与额度预审 (Block Token Hash & `can_allocate`)

- **流式哈希计算**：调度系统在 `can_allocate` 阶段，会将当前请求的 `prefill_tokens` 按照 `PageSize` 转换为**流式哈希 (Streaming Hash)** 序列，并与全局缓存字典进行前缀匹配。
- **额度计算**：借此算出实际需要新分配的 Token 数量（即 `num_dispatched_tokens - num_cached_tokens`），并依此精确评估可用物理显存额度，决定该请求是否准入。

### 2.2 物理页表映射与分配 (Physical Block Mapping & `allocate`)

- **按需分配**：调度系统在 `allocate` 阶段，仅为未命中的增量部分（即 `num_dispatched_tokens - num_cached_tokens`）分配全新的物理页表。
- **全局字典索引**：将分配或命中的物理页表，统一存放到以“流式哈希”为 Key，以 `PageBlockState(物理页表指针, 引用计数)` 为 Value 的全局哈希字典当中，建立全局唯一的块级映射。

## 3. 生命周期分析

物理页表的生命周期由严格的引用计数（Reference Counting）控制，确保数据安全与高效流转。

### 3.1 引用计数的闭环管理

- **Retain (+1)**：在准入与分配阶段，如果发生了流式哈希命中（Cache Hit），或者为新块成功分配了物理页表，字典中对应的引用计数加 1。
- **Release (-1)**：当一个 Sequence 生成完毕（进入 `FINISHED` 状态）或因异常被丢弃（进入 `ABORTED` 状态）而释放资源时，系统必须遍历其持有的所有流式哈希 Key，将字典中对应的引用计数减 1。当计数归零时，该页表转为**可驱逐 (Evictable)** 状态。

### 3.2 缓存命中与两级驱逐策略

- **缓存命中**：在分配页表时，若流式哈希 Key 已存在于字典中且有效，则记录页表命中；否则判定为缺页（Page Fault），触发新页表分配。
- **驱逐策略 (Eviction)**：当 `can_allocate` 或 `allocate` 发现系统物理资源枯竭时，将对原有的抢占驱逐策略进行平滑升级，采用两级防线保障系统运行：
  1. **无感驱逐 (LRU on Evictable)**：优先驱逐不在当前运行批次内的闲置缓存资源。扫描字典中 `引用计数 == 0` 的 Key，采用 LRU（最近最少使用）算法进行回收，直到释放出满足当前请求所需的内存空间。
  2. **兜底抢占 (Active Preemption Fallback)**：如果 LRU 回收后资源依旧不满足（通常发生在 Decode 阶段的显存突增），则触发原有的抢占逻辑，选择低优先级的、正在运行的请求进行页表驱逐和状态回滚，确保高优任务的顺利执行。

## 4. Future Work (未来演进路线)

V1 版本通过极简的块哈希和严密的引用计数搭建了坚实的缓存骨架。未来将在以下方向进行架构演进：

### 4.1 基数树前缀缓存 (Radix-Tree Prefix Caching)

- 取代当前的扁平 HashMap，引入 Radix Tree 数据结构。
- **预期收益**：支持任意长度的 Token 增量匹配（解决半空 Block 无法共享的问题），并原生支持并发请求的 `PENDING` 节点占位机制，彻底消除高并发下的 TOCTOU（查用分离）踩踏。

### 4.2 多级缓存与异步卸载 (Tiered Caching & Offloading)

- 突破单机 HBM（GPU 显存）的物理天花板。
- **预期收益**：当 GPU 显存耗尽且触发 LRU 驱逐时，不直接销毁 KV 数据，而是通过 PCIe/NVLink 异步将其卸载（Offload）至容量更大的 CPU Host Memory，并在未来请求命中时极速 Swap-in。

### 4.3 缓存感知的路由策略 (Cache-Aware Affinity Routing)

- 配合 NanoRoute 等全局网关层进行空间换时间的宏观调度。
- **预期收益**：根据请求的前缀哈希，将共享相同长 Prompt 的请求强制路由至同一台物理节点，从根源上消灭多请求交替调度导致的“单机缓存频繁抖动（Cache Thrashing）”。
